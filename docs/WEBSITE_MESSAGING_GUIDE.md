# Website Messaging Guide

## Goal

The website should feel like the front door of a research product, not a content archive and not an internal ops console.

## Homepage positioning

Primary framing:

> Learn from startup failure like it is a dataset, not a story archive.

Supporting idea:

> Structured cases, grounded analysis, reusable research workflows.

Chinese support line should stay secondary.

## Navigation guidance

Preferred public-facing labels:

- `Explore Cases`
- `Research Hub`
- `Failure Copilot`

Operational labels should stay more muted and more specific:

- `Ops Dashboard`
- `Review Queue`

Avoid broad public labels like:

- 案例库
- 研究入口
- 运营台

Those are understandable internally but weaker for external positioning.

## Metadata rules

Site metadata should stay English-first and factually conservative.

Use language like:

- open-source failure intelligence
- structured startup postmortems
- research workflows
- grounded analysis

Do not use metadata that implies:

- hundreds of cases
- broad market adoption
- enterprise maturity

## Homepage section rules

### Hero

Must answer:

1. What is this?
2. Why is it useful?
3. What stage is it in?

Recommended supporting chips:

- Open-source alpha
- Runnable product
- 40+ published seed cases

### Research Hub section

Frame it as:

- a way to start from a question
- a reusable research entry point
- a faster path than random browsing

### Case explorer section

Frame it as:

- querying a dataset
- filtering research views
- saving structured slices for reuse

## Social and OG tone

Preferred:

- open-source failure intelligence
- structured startup failure research
- grounded failure analysis
- reusable research assets

Avoid:

- graveyard meme project
- fun failed startup stories
- startup cemetery

## Maintenance rule

Whenever README, homepage, and metadata disagree, fix the public-facing language first. Drift in outward messaging damages credibility faster than missing feature copy.
